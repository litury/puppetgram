поддержки и расширяемости кода.

## 1. Общие требования

### 1.1. Иерархия файлов и директорий

- Файлы должны храниться на максимально низком уровне, где они реально используются.
- В корне проекта допускается размещать только системные файлы (`vite.config.ts`, `tsconfig.json`, `package.json` и
  т.п.).
- Структура должна повторять логику предметной области (модули, фичи, компоненты).
- Составные части компонента, должны находиться в директории, соответствующей компоненту.

### 1.2. Названия в терминах предметной области

- Папки и файлы называем так, чтобы их назначение было понятно без чтения кода.
- Используем предметные термины: `user`, `order`, `notification`, а не абстрактные `stuff`, `data1`.

### 1.3. Ограничение количества файлов

- В одной папке не более **5-7 файлов**.
- При превышении — группируем в поддиректории (например, components, api, store, types внутри фичи).

### 1.4. Безопасность

Создавайте в проекте `.gitignore` файл. Добавляйте в него все чувствительные данные, включая, например, `.npmrc` файл.

### 1.5. Документация

Каждый репозиторий должен содержать заполненный `README.md`, описывающий назначение и использование проекта.

Также, в проекте должен быть `ADR.md`, содержащий архитектурные решения. Например, использование хранилищ, протоколов,
библиотек.

## 2. Frontend-приложения

### 2.1. Именование файлов

- **Файлы и директории (JS/TS)** — `camelCase`.
- **Vue-компоненты (.vue)** — `PascalCase` (например, `UserCard.vue`).
- **Композиционные функции** — `useSomething.ts` (только `Composition API`, всегда `script setup` в компонентах).
- **RxJS стримы/сабжекты** — в именах переменных постфикс `$` (`orders$`, `filters$`).
- **Интерфейсы** — `PascalCase` (`IUser`, `IOrderDto`, `IOrderId`).
- **Тесты** — рядом с модулем: `*.spec.ts` (unit), `*.e2e.ts` (e2e).
- **Стили** — глобальные в `src/assets/style/***.css`, локальные внутри компонента в `<style scoped>`.
- **Экспорты** — по умолчанию **именованные**. `default` используем только для `.vue` компонентов.

> В каждой папке — `index.ts` **только** для реэкспорта.

### 2.2. Импорты, алиасы и индексы

- Используем **алиасы** для абсолютных импортов, например:

    - `@/app`, `@/shared`, `@/pages`, **. . .**

- В каждой папке, из которой предполагается импорт **снаружи**, делаем `index.ts` с **именованными экспортами**.
- Относительных импортов уровня `../../../` стоит по возможности избегать — вместо этого заводим alias (`@`) или выносим
  модуль выше.


### 2.3. Разделение файлов по типам. Классическая модульность

| Категория       | Описание                                                                                               |
|-----------------|--------------------------------------------------------------------------------------------------------|
| **app**         | Точка входа приложения (инициализация, маршрутизация, store).                                          |
| **modules**     | Крупные бизнес-модули: `user`, `orders`, `billing`. Каждый — с компонентами, сервисами и store внутри. |
| **shared**      | Общие утилиты, API, базовые UI-примитивы (не зависят от домена).                                       |
| **components**  | Сложные переиспользуемые компоненты (надстройки над ui).                                               |
| **layouts**     | Layout-компоненты (общие шаблоны страниц).                                                             |
| **pages**       | Страницы роутинга (минимум логики).                                                                    |
| **composables** | Vue Composition API хуки (`useAuth`, `useFetch`).                                                      |
| **api**         | HTTP-клиент (axios/fetch), перехватчики, общие DTO.                                                    |
| **config**      | Конфигурации, env, фич-флаги.                                                                          |
| **tests**       | Интеграционные и e2e-тесты.                                                                            |

**Пример структуры (модульный подход):**

```
src/
├── app/ # init, router, store
├── modules/ # бизнес-модули
│ ├── user/
│ │ ├── components/
│ │ ├── services/
│ │ ├── store/
│ │ └── index.ts
│ └── orders/
├── shared/ # utils, api, ui
├── layouts/ # layout-компоненты
├── components/ # сложные переиспользуемые компоненты
├── pages/ # роуты
├── composables/ # хуки Vue
├── config/ # конфиги, env
└── index.ts
```

## 3. UI-библиотеки

### 3.1. Именование файлов библиотек

- Компоненты: `PascalCase` (`AppSelect.vue`).
- Composable: `useXxx` (`useDropdown.ts`).
- Утилиты, классы: `camelCase` (`domUtils.ts`, `basicUser.ts`).
- Типы, enum, интерфейсы: `PascalCase` (`IUser.ts`, `UserRole.ts`).

## 4. Дополнительные рекомендации

- Используйте `index.ts` для удобного экспорта.
- Стремитесь к **чистым и малым компонентам** (Single Responsibility Principle).
- Следуйте правилам именования файлов. Название файла должно отражать его суть/назначение.
- Один тип/интерфейс/класс/функция — один файл.

```ts
// Плохо: один файл экспортирует сразу несколько сущностей. Из названия не понять содержимого файла.

// createUser.ts:
export interface IUser {
  id: number;
  name: string;
}

export function createUser(_id: number, _name: string): BasicUser {
  return new BasicUser(_id, _name);
}

export class BasicUser implements IUser {
  public id: number;
  public name: string;

  constructor(_id: number, _name: string) {
    this.id = _id;
    this.name = _name;
  }
}
```

```ts
// Хорошо: все разбито по отдельным файлам. Имена файлов в точности соответствуют содержимому.

// IUser.ts:
export interface IUser {
  id: number;
  name: string;
}
```

```ts
// createUser.ts:
import { BasicUser } from "./basicUser";

export function createUser(_id: number, _name: string): BasicUser {
  return new BasicUser(_id, _name);
}
```

```ts
// basicUser.ts:
import { IUser } from "./IUser";

export class BasicUser implements IUser {
  public id: number;
  public name: string;

  constructor(_id: number, _name: string) {
    this.id = _id;
    this.name = _name;
  }
}
```

- Не используйте index.ts ни для чего кроме как экспорта содержимого папки. В index.ts не должно быть никакой логики.

```ts
// Плохо: index содержит экспортируемый класс/интерфейс/функцию/тип и т.д.
// Найти класс в структуре документа почти невозможно.

// index.ts:

export class MyAwesomeModule implements IAppModule {
  // ...
}

export * from "./parts";
export * from "./api";
```

```ts
// Хорошо: класс вынесен в отдельный файл и экспортируется через index.ts

// myAwesomeModule.ts:

export class MyAwesomeModule implements IAppModule {
  // ...
}

// index.ts:

export * from "./myAwesomeModule.ts";
export * from "./parts";
export * from "./api";
```
